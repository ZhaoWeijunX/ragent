# 01.Ragent 线程池设计与实现

> 源码入口：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/config/ThreadPoolExecutorConfig.java`

Ragent 在一次对话请求中会同时触发意图识别、多通道检索、MCP 工具调用、记忆加载、LLM 流式输出等多种异步任务。如果全部丢进 `ForkJoinPool.commonPool()` 或共用一个线程池，会出现三类典型问题：

1. **相互抢占**：离线文档分块与在线对话争抢线程，导致 TTFT（首 token 时间）抖动。
2. **无法隔离背压**：某一环节任务堆积时，拖垮整条 RAG 链路。
3. **上下文丢失**：`CompletableFuture.supplyAsync` 切到子线程后，`UserContext`、`RagTraceContext` 等 ThreadLocal 失效，鉴权与链路追踪断裂。

`ThreadPoolExecutorConfig` 的核心思路是：**按业务域拆分线程池、按 CPU 核数动态缩放、用 TTL 传递上下文、按场景选择队列与拒绝策略**。

---

## 1. 设计目标

| 目标 | 说明 |
|------|------|
| 业务隔离 | 在线 RAG、流式输出、记忆摘要、知识库分块各自独立线程池，互不影响 |
| 分层并行 | 子问题 → 检索通道 → 通道内目标，三层并行使用不同池子，避免嵌套争抢 |
| 可观测 | 每个池子有独立 `namePrefix`，线程 dump / 日志可直接定位来源 |
| 上下文传递 | 全部池子经 `TtlExecutors` 包装，保证 traceId、userId 在子线程可用 |
| 与限流对齐 | 对话入口线程数与分布式信号量并发数一致，形成双重约束 |

---

## 2. 全局约定

### 2.1 CPU 动态基准

```java
public static final int CPU_COUNT = Runtime.getRuntime().availableProcessors();
```

线程数不以固定魔法数字写死，而是以 `CPU_COUNT` 为基准用位运算缩放：

| 表达式 | 含义 | 8 核示例 |
|--------|------|----------|
| `CPU_COUNT` | 1× 核数 | 8 |
| `CPU_COUNT << 1` | 2× 核数 | 16 |
| `CPU_COUNT << 2` | 4× 核数 | 32 |
| `CPU_COUNT >> 1` | ½ 核数（向下取整） | 4 |
| `Math.max(2, CPU_COUNT >> 1)` | 至少 2 线程 | 4 |
| `Math.max(4, CPU_COUNT)` | 至少 4 线程 | 8 |

**考虑**：RAG 链路中大量任务是 I/O 密集型（HTTP 调模型、查向量库、调 MCP），线程数可以大于 CPU 核数；但内层并行不宜无限放大，否则会把 Milvus / Embedding 服务打满。

### 2.2 TTL 包装

每个 `ThreadPoolExecutor` 创建后都经过：

```java
return TtlExecutors.getTtlExecutor(executor);
```

项目中的 `UserContext`（登录用户）、`RagTraceContext`（traceId / taskId / 调用栈）基于 Alibaba `TransmittableThreadLocal` 实现。普通 `ThreadLocal` 在 `supplyAsync` 子线程中不可见；TTL 包装后，提交到池子的任务会自动拷贝父线程上下文。

### 2.3 空闲线程回收

所有池子统一 `keepAliveTime = 60s`。仅 `chatEntryExecutor` 额外调用：

```java
executor.allowCoreThreadTimeOut(true);
```

对话低谷期可释放核心线程，减少空转占用；高峰时由限流器控制重新扩容。

### 2.4 Bean 注入方式

Spring 按 **方法名 = Bean 名** 注入。业务类中字段名与 Bean 名一致即可精确匹配，例如：

```java
private final Executor ragContextExecutor;  // → ragContextExecutor Bean
```

## 3. 队列与拒绝策略选型

### 3.1 两种工作队列

| 队列类型 | 行为 | 适用场景 |
|----------|------|----------|
| `SynchronousQueue` | 不缓存任务；提交时必须有空闲线程接手，否则尝试扩容到 max，再不行触发拒绝策略 | 在线热路径：希望并发上限可预期，拒绝无界堆积 |
| `LinkedBlockingQueue(cap)` | 有界缓冲，队列未满时 core 线程忙不过来也不会立刻扩容到 max | 允许短暂突发、或任务提交方需要兜一层缓冲 |

### 3.2 两种拒绝策略

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `CallerRunsPolicy` | 由提交任务的线程自己执行 → 自然降速，任务不丢 | 检索、意图、MCP 等「宁可慢一点也不能静默丢任务」的路径 |
| `AbortPolicy` | 直接抛 `RejectedExecutionException` | 入口限流、流式连接、离线分块等「满了就快速失败」的路径 |

## 4. 九个线程池详解

以下以 **8 核 CPU、默认配置** 为例列出参数；实际值随部署机器变化。

### 4.1 `chatEntryExecutor` — 对话执行入口

| 参数 | 值 |
|------|-----|
| core / max | `rag.rate-limit.global.max-concurrent`（默认 **50**） |
| 队列 | `SynchronousQueue` |
| 拒绝策略 | `AbortPolicy` |
| 特殊 | `allowCoreThreadTimeOut(true)` |

**调用位置**：`ChatQueueLimiter.enqueue()`。用户通过 SSE 发起对话后，先经 `FairDistributedRateLimiter` 排队拿分布式许可，再在 `chatEntryExecutor` 中执行真正的 RAG 主流程。

**设计意图**：

- 线程数与 Redis 信号量许可数 **1:1 对齐**，避免「拿到了许可但没有线程执行」或「线程空闲但许可已耗尽」的不一致。
- `SynchronousQueue` + 固定大小：最多同时有 N 个对话在处理，无排队缓冲。
- `AbortPolicy`：池满时抛异常，`ChatQueueLimiter` 捕获后向前端返回「系统繁忙」，与限流超时走同一 reject 流程。
- 这是全链路 **最外层闸门**，必须快速失败，不能用 `CallerRunsPolicy`（会在 Tomcat 线程上跑完整 RAG，阻塞接入层）。

**配置项**（`application.yaml`）：

```yaml
rag:
  rate-limit:
    global:
      enabled: true
      max-concurrent: 50      # → chatEntryExecutor 线程数
      max-wait-seconds: 20
      lease-seconds: 600
```

---

### 4.2 `intentClassifyExecutor` — 子问题意图并行

| 参数 | 值（8核） |
|------|-----------|
| core / max | 8 / 16 |
| 队列 | `SynchronousQueue` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`IntentResolver.resolve()`。问题改写可能拆出多个子问题，每个子问题并行做意图分类（embedding 匹配 + LLM 分类）。

**设计意图**：

- 子问题数量通常 ≤ 5，不需要超大池；core = 1×CPU 足够覆盖常见并行度。
- max = 2×CPU 应对改写结果较多的边界情况。
- 意图分类失败会降级为空意图列表，因此用 `CallerRunsPolicy` 保证任务最终被处理。

---

### 4.3 `ragContextExecutor` — 子问题级上下文构建

| 参数 | 值（8核） |
|------|-----------|
| core / max | 32 / 32（固定） |
| 队列 | `SynchronousQueue` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`RetrievalEngine.retrieve()`。每个子问题并行执行「KB 多通道检索 + MCP 工具调用 + 上下文格式化」。

**设计意图**：

- 这是检索阶段的 **最外层并行**，子问题之间完全独立，线程数设为 4×CPU。
- core == max：固定大小池，防止无限制创建线程。
- 每个子问题内部还会触发 `ragRetrievalExecutor` 和 `mcpBatchExecutor`，因此这一层需要足够宽，但不能无限大。

---

### 4.4 `ragRetrievalExecutor` — 检索通道级并行

| 参数 | 值（8核） |
|------|-----------|
| core / max | 32 / 32 |
| 队列 | `SynchronousQueue` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`MultiChannelRetrievalEngine.executeSearchChannels()`。并行执行所有启用的检索通道（如向量全局检索、意图定向检索等）。

**设计意图**：

- 通道数量一般 2～4 个，但每个通道耗时较长（向量检索 + Rerank），需要与 `ragContextExecutor` 同级的大池保证多通道同时跑。
- 与 `ragContextExecutor` 参数相同是刻意的：同一请求可能同时有多个子问题 × 多个通道，两层并行峰值叠加由各自的 `SynchronousQueue` 限制。

---

### 4.5 `innerRetrievalExecutor` — 通道内部并行检索

| 参数 | 值（8核） |
|------|-----------|
| core / max | 16 / 32 |
| 队列 | `LinkedBlockingQueue(100)` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`IntentDirectedSearchChannel`、`VectorGlobalSearchChannel` → `AbstractParallelRetriever.executeParallelRetrieval()`。对多个意图节点或 collection 并行发起向量检索。

**设计意图**：

- **最内层并行**，目标数可能较多（多个意图叶子 × 多个 collection）。
- 使用 **有界队列 100**：允许短暂突发（比如一次命中 10+ 意图节点），避免瞬间打满 Milvus；队列满后 `CallerRunsPolicy` 在通道线程上降级执行。
- core = 2×CPU、max = 4×CPU：比通道级略小，体现「内层并发应有节制」的原则，保护下游向量库。

---

### 4.6 `mcpBatchExecutor` — MCP 工具并行调用

| 参数 | 值（8核） |
|------|-----------|
| core / max | 8 / 16 |
| 队列 | `SynchronousQueue` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`RetrievalEngine.executeMcpTools()`。对多个 MCP 工具并行 `callTool`。

**设计意图**：

- MCP 调用是典型 I/O 等待（外部 HTTP / stdio），core = 1×CPU、max = 2×CPU 与 `intentClassifyExecutor` 一致。
- 单个工具失败会构造 error result 降级，不应因线程池拒绝而丢工具调用；故用 `CallerRunsPolicy`。

---

### 4.7 `memoryLoadExecutor` — 对话记忆并行加载

| 参数 | 值（8核） |
|------|-----------|
| core / max | 4 / 8 |
| 队列 | `LinkedBlockingQueue(200)` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`DefaultConversationMemoryService.load()`。并行加载「历史摘要」和「近期消息列表」两个 DB 查询。

**设计意图**：

- 每次 `load()` 仅提交 **2 个任务**，池子不需要很大；`Math.max(2, CPU/2)` 保证低核数机器也有基本并行能力。
- 记忆加载在 RAG 主链路前端，延迟敏感但任务数固定；`CallerRunsPolicy` 确保极端情况下降级为同步加载，不返回空记忆。

---

### 4.8 `memorySummaryExecutor` — 对话摘要异步压缩

| 参数 | 值（8核） |
|------|-----------|
| core / max | **1** / 4 |
| 队列 | `LinkedBlockingQueue(200)` |
| 拒绝策略 | `CallerRunsPolicy` |

**调用位置**：`JdbcConversationMemorySummaryService.compressIfNeeded()`。assistant 回复后 **异步** 触发 LLM 摘要压缩。

**设计意图**：

- **后台任务**，不阻塞用户对话响应。
- core = 1：同一时刻只压缩一个会话，避免大量 LLM 摘要调用同时发出（成本高、易触发模型限流）。
- max = 4、队列 200：允许多个会话排队等待压缩，但不会无限并发。
- 失败仅打日志，不影响主链路。

---

### 4.9 `modelStreamExecutor` — LLM 流式读取

| 参数 | 值（8核） |
|------|-----------|
| core / max | 4 / 8 |
| 队列 | `LinkedBlockingQueue(200)` |
| 拒绝策略 | **`AbortPolicy`** |

**调用位置**：`AbstractOpenAIStyleChatClient.doStreamChat()` → `StreamAsyncExecutor.submit()`。在独立线程中阻塞读取 SSE 响应流，解析 token 回调给前端。

**设计意图**：

- 每个流式对话占用 **一个线程长时间阻塞**（直到流结束），线程数直接决定最大并发流数。
- `AbortPolicy`：池满时拒绝新流式任务，防止无限制创建长连接线程拖垮 JVM。
- 与 `chatEntryExecutor` 配合：入口限流控制「开始处理的对话数」，流式池控制「同时进行流式读取的连接数」。两者参数独立，可按机器资源分别调优。

---

### 4.10 `knowledgeChunkExecutor` — 知识库文档分块

| 参数 | 值（8核） |
|------|-----------|
| core / max | 4 / 8 |
| 队列 | `LinkedBlockingQueue(200)` |
| 拒绝策略 | **`AbortPolicy`** |

**调用位置**：`KnowledgeDocumentScheduleJob` 定时任务。扫描到期文档后，将分块刷新处理提交到线程池。

**设计意图**：

- **离线/定时任务**，与在线 RAG 完全隔离。
- 调度线程（`@Scheduled`）只负责扫描和投递；实际分块在池子中执行。
- `AbortPolicy` + `RejectedExecutionException` 捕获：队列满时释放分布式锁，避免任务卡死；下次调度重试。
- 参数与 `modelStreamExecutor` 相同，体现「非核心路径、中等并发、快速失败」的统一策略。

## 5. 一次对话中的线程池调用层次

```
HTTP 请求
  └─ ChatQueueLimiter
       └─ [chatEntryExecutor] RAG 主流程
            ├─ [memoryLoadExecutor] 并行加载摘要 + 历史（2 任务）
            ├─ [intentClassifyExecutor] 子问题意图并行
            ├─ [ragContextExecutor] 子问题上下文并行
            │    ├─ [ragRetrievalExecutor] 多通道并行
            │    │    └─ [innerRetrievalExecutor] 意图/Collection 并行
            │    └─ [mcpBatchExecutor] MCP 工具并行
            ├─ [modelStreamExecutor] LLM 流式 SSE 读取
            └─ (异步) [memorySummaryExecutor] 对话摘要压缩
```

**关键原则**：

1. **由外到内**：越靠近用户请求，越要严格限流（`chatEntryExecutor`）；越靠近下游存储，越要节制并发（`innerRetrievalExecutor`）。
2. **横切隔离**：意图、MCP、记忆、流式、分块各有独立池，某一环节慢不会占满其他环节的线程。
3. **同步等待**：各层通过 `CompletableFuture.join()` 等待子任务完成，线程池嵌套时外层线程会阻塞等待，因此内层池子必须有独立容量。

## 8. 调优建议

### 8.1 对话并发

优先调整 `rag.rate-limit.global.max-concurrent`，它同时影响 Redis 信号量和 `chatEntryExecutor` 大小。增大前需评估：

- JVM 堆内存（每个对话会持有 SSE 连接、检索上下文）
- `modelStreamExecutor` 能否承接更多流式线程
- 下游 Milvus / Embedding / LLM 服务的 QPS 上限

### 8.2 检索延迟

若向量检索 P99 升高，优先 **缩小** `innerRetrievalExecutor` 的 max 或队列容量，而不是扩大 `ragRetrievalExecutor`。内层并发是直接打到 Milvus 的请求数。

### 8.3 观测手段

```bash
# 线程 dump 中按 namePrefix 过滤
jstack <pid> | grep -E "rag_context|rag_retrieval|chat_entry|model_stream"
```

配合 `RagTraceContext` 链路日志，可定位「哪个池子的线程在执行慢任务」。

### 8.4 不建议的做法

| 做法 | 原因 |
|------|------|
| 合并所有池为一个 `commonExecutor` | 离线分块与在线对话互相抢占 |
| 全部改用 `CallerRunsPolicy` | Tomcat 线程可能被拉去跑完整 RAG，接入层阻塞 |
| 全部改用无界 `LinkedBlockingQueue` | 任务无限堆积，OOM 风险 |
| 去掉 TTL 包装 | 子线程丢失 userId / traceId，鉴权和追踪失效 |

---

## 9. 小结

Ragent 的线程池方案可以概括为 **「分池、分层、分策略」**：

- **分池**：9 个独立 Bean，按业务域隔离资源。
- **分层**：子问题 / 通道 / 通道内目标三层并行，各用不同池，保护下游。
- **分策略**：在线热路径 `SynchronousQueue` + 背压；后台/缓冲场景有界队列；入口和流式 `AbortPolicy` 快速失败，检索链路 `CallerRunsPolicy` 保证不丢任务。

理解各池子在 RAG 流水线中的位置，是排查「系统繁忙」「检索变慢」「流式发不出 token」等问题的第一步。
